🔴 발견된 문제점
1. Operator Cross-Namespace 프록시
Medium

Operator는 polyon-operator namespace에 정상 배포 (https://operator.cmars.com)

  • Console nginx: /api/sentinel/*, /api/setup/* → polyon-operator:1110
  • Console은 polyon namespace → 다른 namespace의 short name DNS 미해석
  • 확인 필요: nginx.conf에서 Operator 프록시가 FQDN 사용하는지 확인
  • 해결: polyon-operator.polyon-operator.svc.cluster.local:1110 사용
2. /computers API 경로 불일치
High

Dashboard StatCards가 apiFetch('/computers') 호출 → 404

  • Console dashboard.ts: apiFetch('/computers') = /api/v1/computers
  • Core 라우트: /api/v1/domain/computers (domain prefix 필요)
  • 영향: Dashboard 컴퓨터 수 표시 실패
  • 해결: Console에서 /domain/computers로 변경 또는 Core에 /computers alias 추가
3. Secret 비밀번호 불일치
Medium

polyon-common-secret.POLYON_DB_PASSWORD ≠ polyon-db-secret.POSTGRES_PASSWORD

  • common-secret: 6v9gjOQxf7WDhdhlBcE7s65x (Operator 생성)
  • db-secret: UURlOTFlcRJOoMlEKdIMZcbR (실제 PG 비밀번호)
  • pgweb은 db-secret 직접 참조로 해결됨
  • 영향: Core가 common-secret 참조 시 DB 연결 실패 가능
  • 해결: Operator에서 DB Secret 생성 시 common-secret과 동기화
4. nginx.conf 이중 관리
Medium
  • 이미지 내장: Docker 빌드 시 console/nginx.conf → /etc/nginx/conf.d/default.conf
  • ConfigMap: polyon-console-nginx가 같은 경로에 마운트하여 오버라이드
  • 현재 두 파일 내용 동일 (동기화됨)
  • 위험: 이미지만 재빌드하고 ConfigMap 미수정 시 불일치 발생
  • 해결: 이미지에서 nginx.conf 제거하고 ConfigMap 단일화, 또는 빌드 스크립트에서 자동 동기화
5. Containers 페이지 API (K8s 전환)
Medium
  • /api/v1/pods, /api/v1/pods/{name}/logs, /api/v1/pods/{name}/restart
  • /api/v1/services/topology, /api/v1/containers/volumes
  • Core의 containers.go가 Docker API 기반 → K8s에서 작동 여부 미확인
  • 영향: Containers 페이지 3개 (목록/토폴로지/리소스) 전부 불안정
